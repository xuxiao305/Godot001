# 0001 — 采用"内在发动机派"作为角色控制架构

## Context

3C 原型设计第一版（2026-05-24）选择**速度覆盖派**作为角色控制架构，论据为"行业标准、丝滑、保留物理交互"。原文档明确否决了"路径 ①：纯动力学（force/impulse）"，理由是"太粘"。

## Decision

**架构反转：改采用"内在发动机派"**（force/impulse-based，每帧不覆盖 `linear_velocity`）。

玩家所有操作（移动、跳跃、空中转向）都转化为施加在 Box2D dynamic body 上的力或冲量，由 Box2D 物理解算演化速度。角色自发的力与场景施加的力（爆炸、推、碰撞）处于同一本体论层级。

## Rationale

- 项目愿景"2D 神秘海域 setpiece"要求物理反馈在所有场景自然涌现（爆炸推飞、撞凸起、动平台、塌方），而非靠特例代码挂载在覆盖架构上
- 速度覆盖派每次涌现都要写"短暂关闭覆盖"的特例，复杂度随 setpiece 数量线性增长
- 接受"非 Celeste 输入感"代价；用**高启动加速度**（≈ 80 m/s², 0.1 s 到顶速）补偿响应感，用**摩擦力可调**保留惯性感

## Consequences

- 3C 文档 §2（设计哲学）、§2.2（定位图）、§2.3（架构选择表）、§4.2-4.4（运动 / 跳跃参数）、§5.1（执行顺序）需要全面重写
- §1.3 成功标准重新表述：从"输入丝滑"改为"物理反馈一致 + 高响应启动 + 可感知惯性"
- §2.3 注释"v1 留接口不实现"的"短暂关闭速度覆盖"机制**不再需要** —— 物理反馈始终生效
- §2.2 项目定位左移：从"≈ Hollow Knight ↔ Ori"改为"≈ Trine / HL2 / 偏向 LIMBO 但启动响应强"
- 后续家族 B（破坏）/ C（流体）/ D（武器）所有与角色的物理交互**免费可用**，无需架构层对接
