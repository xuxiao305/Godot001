## 💧 I-001 统一物理下的粒子水体

**状态：** ✅ 契合新方向（涌现派）。是触发 [ADR-0012 项目方向转向涌现派](adr/0012-pivot-to-emergent-simulation.md) 的导火索。优先级提升为核心探索方向之一。

**核心想法：** 在统一的物理机制下，水由大量小粒子模拟。所有 block 与水粒子是同一物理本体——能互相碰撞、互相施力。水具备浮力，每个 block 根据自身密度决定漂浮还是下沉（包括子弹弹壳之类小物件）。

**潜在玩法：**
- 物体落水产生真实排开
- 木块漂、铁块沉，不需要"是否浮"的标记，由物理统一决定
- 子弹壳落水溅起，弹壳本身也会被水流推走
- 沉船时船舱进水、物品被冲散

**与已锁定决策的关系（2026-05-25 转向后）：**
- ✅ **完美契合** [[项目总览]] §2.1 系统 > 编排（涌现派）
- ✅ **完美契合** §2.5 物理统一优先——是统一物理的极致形式
- ✅ **完美契合** "内在发动机派"（[[platformer-physics-tech-stack]]）——水粒子和角色都是 dynamic body，统一规则
- ✅ **已解除** 原 §3 / §8 的"❌ 真流体"红线，详见 [ADR-0012](adr/0012-pivot-to-emergent-simulation.md)

**待解决问题（进 brainstorming 时必答）：**
- 性能：Box2D 同屏多少 dynamic body 不卡？1k？5k？决定"水"的视觉密度
- 水粒子之间需要相互作用力（粘性、表面张力）吗？还是仅靠碰撞？纯碰撞水体看上去像"会流的沙"，没水的样子
- 视觉渲染：粒子怎么合成水面？Metaball？Marching squares？
- 是否完全放弃"高度场水"备选路线？还是作为性能 fallback 保留

**最大风险：** 性能。粒子数与视觉真实度强相关，Box2D 同屏粒子上限将决定方案可行性。建议第一步就是做粒子数压力测试 demo，而不是先做美术。

## 🛠️ I-002 零件组装载具

**核心想法：** 玩家拾取/获取零件，自行组合成载具。

**渐进示例：**
1. 2 轮 + 1 板 = 滑行车（无动力，靠斜坡 / 推力）
2. + 动力源 = 车
3. + 武器 = 战车

**与已锁定决策的关系：**
- ✅ **完美契合** §3 Box2D 白送能力（关节系统 + 车辆例程）
- ✅ **完美契合** §2.5 "物理统一优先"——组装规则纯物理（焊接关节 + 摩擦轮 + 力施加）
- ✅ **扩展** §4 家族 E（载具）——不是新家族，是"玩家组装"角度的新子原型
- ✅ **可与** §4 家族 B（破坏）联动——战车被打散，零件散落可回收

**待解决问题（进 brainstorming 时必答）：**
- 组装 UX：拖拽吸附？还是放近距离自动焊接？
- 零件接口规范：怎么定义"能接的位置"？预设吸附点 vs 自由焊接
- 动力源逻辑：作用在哪个 body？怎么传到轮子？
- 战车上的武器朝向跟车体？还是独立瞄准？
- 解体规则：什么强度的冲击会让焊接断？（复用 [[2026-05-24-destruction-prototype-design]] 的关节断裂逻辑）

**最大风险：** UX 设计深坑。"拼装感"做不好就是玩具，做好了是 demo 杀器。建议先做"硬编码的预组装车 + 可被破坏"，再升级到"玩家自由组装"。

## 💦 I-003 水体的力响应

**核心想法：** 水不是被动的体积，会响应外部冲力——
- 物体落水：根据质量与速度产生溅起的水花
- 爆炸：水面被冲击波推开，形成短暂凹陷与浪
- 子弹/炸弹入水：自动溅水花，不需要为每个武器手写

**与已锁定决策的关系：**
- ✅ **天然契合** §3 ⚠️ 替代方案"流体 → 高度场水"——高度场天然支持局部高频扰动
- ✅ **是** §4 家族 C "高度场水 + 浮力" 子原型的合理扩展，不是新分支
- ✅ **不与** I-001 互斥——即使保持高度场水路线也能实现
- ✅ **与** §4 家族 D（武器/爆炸）联动，统一接口

**待解决问题（进 brainstorming 时必答）：**
- 冲击 → 高度场扰动的传函：能量 → 波幅 / 波长 / 衰减率怎么映射
- 水花视觉：粒子系统？纹理动画？高度场峰值上的临时粒子喷发？
- 性能：高度场分辨率 vs 视觉细腻度的权衡
- 是否需要 "水深感"——浅水溅多，深水溅少？

**最大风险：** 调参深坑（什么力 → 什么水花是"对的"）。但下限可控——做出来就是加分项，做不好也不会破坏其他系统。

## 维护规则

- 新想法直接加 `## I-XXX 标题` 段落
- 每个想法**必须**带"与已锁定决策的关系"段落
- 升级为正式原型时：
  1. 在本文件标记 `**状态：** 已升级 → [[spec 文件名]]`，保留段落
  2. 在 [[项目总览]] §4 家族表或 §6 路线图添加条目
  3. 走 brainstorming → spec → plan
- 明确放弃的想法：标记 `**状态：** 放弃 — <原因>`，保留段落作为历史
